Minimum functionalities

Minimum functionality refers to a set of requirements that a system needs so it can meet the needs of its users and stakeholders.

Using the MoSCoW grading (must have, should have, could have) for how important things are, we cover the main features that a system must have. Some requirements are 'should have'. These are requirements which provide local authorities and end users with options. They will not stop the system from working as it should, but they will make it useful for more users.

A lot of these features are supported by the standard and schema.

  • Data Standard
  • Data Schema

EHCP process visibility and final version

Your system must give transparency to parents, carers, families, caseworkers and other contributors so they can monitor and track the status of EHCPs. This will show which stages are complete, and which require input.

Your system must allow Special Educational Needs and Disabilities (SEND) teams and health professionals to track cases by statutory deadlines.

Your system should allow for a final version of the EHCP that can be:

  • approved by all relevant people
  • locked down to avoid further editing
  • submitted within the 20-week period

It must allow contributors to view relevant sections of the EHCP after it is submitted.

It must produce a Plan that complies with the standardised template.

Access control

Users must have individual accounts to access the system. Role-based access will vary across the EHCP process, but it must be clear who has updated the EHC record and when.

All contributors must have view access throughout the process.

Data must not be overwritten, including parents, carers and self-contributors, advocate for self, or over-16s and ownership.

Your system must have an audit trail.

Image and video contributions

Your system must allow for contributors to upload images, video and audio files to enable representation of a child/young person.

Parent / carer consent form capturing consent for multimedia - specify holding this consent and signature.

Images and videos should be held in standard formats to enable use on a wide variety of devices.

Login, error prevention and recovery

Your system should include:

  • multi-factor authentication for setting up username and password and for logging in
  • ways for users to resolve sign-in issues themselves
  • a password recovery function
  • timeout, logging users out of a screen left inactive for a certain time
  • the ability for users to see an individual's previously published EHCP(s)

Saving

Your system must give the ability to save updates. This includes:

  • auto-save for written text and uploaded documents to prevent users from losing contributions
  • a save button that saves updates in the back end of the system and notifies users when updates are saved successfully
  • a save button at each stage

Storage and ability to function on different devices

Your system must have a minimum storage capacity of xxx so that they can operate appropriately.

It must work on different devices, including:

  • desktop
  • tablet
  • mobile

System errors

  • provide clear error messages when a user inputs something incorrectly – giving them the ability to solve the mistake themselves
  • A clearly defined escalation process in the situation where an error message cannot resolve the users input

Notifications

Notifications must:

  • have different language for different notifications depending on who the notification is for. This should be determined between LAs and suppliers
  • consider preferred method for communication
  • have some sort of identifier to identify child, young person
  • be sent at key stages of the process

In terms of the content of the notifications, they should notify users through an on-screen message:

  • that a task has been completed – items selected, text saved, or upload successfully made via an on-screen message.

Notifications must not be generic and should be specific. They should always refer to a specific EHCP rather than "you have something to complete" - they need an identifier that doesn’t contravene GDPR.

Notifications need to give users the option of preferred method of communication.

Accessibility and usability

The system should:

  • provide the same digital experience across all devices, browsers and with assistive technology
  • be tested with and then support screen readers, high contrast mode and others
  • meet WCAG x.x for various software / OSs - testing would vary
  • provide the child/young person, parents and carers with offline experiences so they do not rely on digital system access to complete EHCPS

The system must provide the CYP, parents and carers with offline experiences so they do not rely on digital system access to engage.

Import/export functionality and formats

The system must support:

  • sharing EHC records in a format compliant with the templates and the data standard
  • creating or storing a plan which complies with the standard EHCP template
  • downloadable and shareable formats despite connection
  • controlling what can be imported and exported by who

Wider reporting needs include SEN2, finance reporting – which require data to be imported and exported to (specification)
