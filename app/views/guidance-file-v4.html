Education Health Care Plan (EHCP) minimum functionalities

This information has been produced to create a benchmark for the minimum requirements to build a system that can successfully manage Education, Health, and Care Plans (EHCPs).

It has been co-produced with suppliers and local authorities (LAs) to determine exactly what must be included.

The purpose of which is to give LAs and suppliers the right information when starting to digitise their own EHCPs.

What do we mean by minimum system requirements?

Minimum system requirements refers to a set of requirements that a system needs so it can meet the needs of its users and stakeholders.

Using the MoSCoW grading (must have, should have, could have) for how important things are, we cover the main features that a system must have. Some requirements are 'should have'. These are requirements which provide local authorities and end users with options. They will not stop the system from working as it should, but they will make it useful for more users.

Term This is a mandatory requirement
Must This is a mandatory requirement
Should Something that is a strongly advised requirement but not essential
Could Something that is a possibility

Definitions

EHCP – Education health and care plan

CYP – Child or young person

Out of scope

All of these requirements are the minimum required from a system. Any functionality above and beyond should be discussed agreed by the LA and supplier (or between those in the LA if they are building it themselves). This does come with a risk that should an EHCP be transferred between systems then some of this additional functionality might not transfer as it goes above the minimum needed.

System requirements

Visibility of EHCP process

The system must give transparency to parents, carers, families and over-16s so they can monitor, track and see the status of their application and plan in the EHCP process.

The system must allow contributors to keep view access of the own contribution even after submitting their advice, while they still have contact with the CYP. There will be local autonomy around different contributors seeing other contributions. All contributors must be able to see the finished plan, if they are still in contact with the CYP.

The system must allow professionals, including health and SEND teams, to track cases by statutory deadlines and additional deadlines.

The system must capture the high-level timeline of the EHCP process. These are detailed in the Send Code of Practice.

The system must produce a Plan that complies with the standardised template and is downloadable and easily accessible for the intended user.

Digital co-production and access control

The system must define roles that can be assigned to different users depending on their level of access and when their input is needed.

According to the defined roles in the system, users must have visibility of their contributions for as long as it is needed.

The system must not allow any parent or CYP to amend another contributor's advice. Additionally, the system must not allow the data of individual parents, carers and self-contributors, advocate for self for over 16s to be overwritten. Only a professional should have access to amend another contributor’s advice if the original professional no longer has access.

The system must allow for supporting files to be uploaded for the EHCP process.

The system must include an audit trail of all actions that result in a change of data; recording time, date, user, and change in record. It must not be able to edit the audit trail.

The system must allow parent/carer consent to be captured when uploading image, videos, or any representation of the CYP.

The system must only give people access to the appropriate information for their role.

The system must have the ability to assign users as appropriate to the stages where their input is needed.

The system must not allow the plan to be editable.

Save, troubleshooting and recovery

The system must have a save or auto-save functionality to prevent users from losing contributions when using the system and notify users that their contributions have been saved.

The save function should be present at each stage of the application in accordance to the standard templates.

The system must have a recovery function for credentials for ease of access into the system in the event of needing to regain access.

There must be a route to accessible troubleshooting at the appropriate level for each user group that allows them to resolve any issues when using the system.

The system must have the functionality for data to be backed up.

The system must timeout if left open for a certain period of time. This should be aligned with GOV.UK standards – 15 minutes of inactivity.

Notifications

The system must allow notifications and alerts to be configured that prompt users to provide input and enter their contributions at relevant stages. These notifications will be different for each type of user.

The system must allow email notifications to be sent and could have an option for SMS alerts.

The alerts and notifications must be specific and have an identifier to the case they relate to.

The system should notify users according to a particular application rather than simply state: "you have something to complete".

Compliance/accessibility

The system must have consistency across devices and allow users the ability to complete tasks in the same way.

The system must comply and meet the WCAG 2.2 standard across all devices, browsers and operating systems.

The system must support the use of screen readers, high contrast mode and other accessibility tools.

The system must follow regulations such as GDPR to ensure data privacy and protection.

The system must provide the CYP, parents and carers with offline experiences so they do not rely on digital system access to engage.

Transfer mechanism

The system must allow EHC records to be imported and exported in a format that is compliant with the data standard and plan templates.

The system must control access as to who and what can be imported and exported.

The system must allow the import and export of attached documentation and artefacts.
